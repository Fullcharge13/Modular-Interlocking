# 공포(栱包) 에이전트 하네스

> 컨텍스트 하중을 4층으로 분산하여 토큰 낭비 없이 작동하는 Python 개발 시스템.
> 공포가 못 하나 없이 수십 톤을 버티듯, 이 시스템은 하드코딩 없이 복잡도를 지탱합니다.

---

## 기둥 (런타임 원칙)

이 파일은 모든 세션의 시작점입니다. 아래 규칙은 예외 없이 적용됩니다.

### 1. 4층 구조를 항상 인식할 것

```
Layer 1: rules/     → 결구  — 불변 규칙, 재해석 금지
Layer 2: skills/    → 주두  — 도메인 지식 분산, 최소 컨텍스트
Layer 3: agents/    → 첨차  — 독립 실행, 실패 격리
Layer 4: hooks/     → 공포  — 층간 검증 게이트
```

### 2. 컨텍스트 예산 (주두 원칙)

- 단일 에이전트 호출에 **전체 히스토리를 전달하지 않는다**
- 서브에이전트에게는 **Spec + Interface만** 전달한다
- 컨텍스트가 **50%** 에 도달하면 `skills/compact-trigger`를 즉시 실행한다
- 컨텍스트가 **70%** 에 도달하면 작업을 중단하고 압축 후 재개한다

### 3. 에이전트 위임 원칙 (첨차 원칙)

| 태스크 유형 | 위임 대상 |
|---|---|
| 기능 구현 계획 | `agents/planner` |
| Python 코드 리뷰 | `agents/python-reviewer` |
| TDD 사이클 강제 | `agents/tdd-guide` |
| 컨텍스트 상태 감시 | `agents/context-guardian` |

### 4. 검증 게이트 (공포 원칙)

모든 코드 변경은 다음 게이트를 통과해야 한다:
1. **결구 게이트**: 테스트가 먼저 작성되었는가?
2. **주두 게이트**: 컨텍스트 예산 내에서 작업하고 있는가?
3. **첨차 게이트**: 실패가 격리되어 있는가?
4. **공포 게이트**: 전체 테스트 스위트가 통과하는가?

## 모델 전략

```
기본: claude-sonnet    → 일상 코딩, 리뷰, TDD 사이클
심화: claude-opus      → 아키텍처 설계, 복잡한 디버깅
서브에이전트: haiku    → 단순 검색, 파일 읽기, 포맷팅
```

```bash
/context-status

/compact-now
```

## 이 시스템에 대해

- **철학**: 공포(栱包) — 층층이 맞물려 하중을 분산, 못 없는 결합
- **대상**: Claude Code + Python 개발
- **핵심 문제**: 컨텍스트 폭발 → 토큰 낭비 → 품질 저하
- **핵심 해법**: 능동적 컨텍스트 감시 + 계층적 분산
