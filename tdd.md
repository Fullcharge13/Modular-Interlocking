# TDD 철의 법칙 (Iron Law of TDD)

> **결구 원칙**: 이 규칙은 불변입니다. "이번 한 번만"은 없습니다.

---

## 핵심 법칙

> **프로덕션 코드는 실패하는 테스트 없이 작성하지 않는다.**

공포의 결구처럼, 테스트는 코드가 미끄러지지 않도록 홈을 파는 행위다.
홈 없이 나무를 올리면 반드시 미끄러진다.

---

## RED → GREEN → REFACTOR 사이클

### RED
```
1. 단 하나의 동작을 검증하는 최소 테스트 작성
2. 테스트 이름은 동작 기술 (메커니즘 아님)
   ✅ test_user_cannot_login_with_wrong_password
   ❌ test_authenticate_function
3. 실행 → 반드시 실패해야 함
4. 실패 이유가 "올바른 이유"인지 확인
```

### GREEN
```
1. 테스트를 통과시키는 가장 단순한 코드 작성
2. 추가 파라미터, 설정, 미래 기능 금지
3. 전체 스위트 실행 → 모두 통과
```

### REFACTOR
```
1. 동작 변경 없이 코드 정리
2. 매 변경 후 전체 스위트 재실행
3. 모두 통과 확인 후 커밋
```

## Python TDD 도구 스택

```python
# 기본 테스트 프레임워크
pytest >= 7.0

# 커버리지 측정
pytest-cov

# 목킹
pytest-mock
unittest.mock (표준 라이브러리)

# 비동기 테스트
pytest-asyncio

# 빠른 실행 (변경된 파일만)
pytest-xdist
```

### 실행 명령
```bash
# 기본 실행
pytest

# 커버리지 포함
pytest --cov=src --cov-report=term-missing

# 실패한 테스트만 재실행
pytest --lf

# 변경된 파일 감시
pytest-watch
```

|---|---|

| 유틸리티 | 75% |
| 전체 | 80% |

## 합리화 규칙 (Anti-Rationalization)

다음 변명은 모두 무효다:

| 변명 | 응답 |
|---|---|
| "이미 코드를 작성했다" | 매몰비용 오류. 삭제하고 TDD로 재작성. |
| "나중에 테스트 추가할게" | 나중의 테스트는 "이게 뭘 하는가"를 검증한다. TDD는 "뭘 해야 하는가"를 정의한다. |
| "너무 단순해서 테스트 불필요" | 단순한 코드도 부러진다. 테스트는 30초다. |
| "테스트 작성이 어렵다" | 테스트하기 어려운 코드 = 사용하기 어려운 코드. 설계 신호다. |
| "이번 한 번만 예외" | 그 한 번이 결구를 부순다. |

## 버그 수정 규칙

버그를 수정하기 전에 반드시:
1. 버그를 재현하는 실패 테스트 작성
2. 테스트가 올바른 이유로 실패하는지 확인
3. 루트 원인 수정 (증상 아님)
4. 테스트 통과 확인
5. 회귀 방지 확인

## Git 커밋 규칙

```
RED:      git commit -m "test: [동작 설명] - failing"
GREEN:    git commit -m "feat: [구현 설명] - passing"
REFACTOR: git commit -m "refactor: [정리 내용]"
```
