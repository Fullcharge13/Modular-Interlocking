# Compact Trigger (압축 트리거)

> **주두 원칙**: 이 skill은 지붕이 처지기 전에 활처럼 하중을 양쪽으로 나눈다.
> 첨차(檐遮)가 하중을 분산하듯, 컨텍스트를 압축하여 다음 작업이 가볍게 시작되게 한다.

---

## 역할

컨텍스트 압축이 필요한 **정확한 시점**을 판단하고, 최소한의 손실로 압축을 실행한다.
무조건 압축하지 않는다 — 잘못된 시점의 압축은 진행 중인 작업을 망친다.

---

## 발동 조건 (context-sentinel에서 위임)

```
✅ 압축 실행 → 지금이 맞다
   - 컨텍스트 50%+이고 현재 태스크 사이 경계
   - 연구/탐색 단계 완료 직후
   - 마일스톤(기능 완성, 테스트 통과) 직후
   - 실패한 접근을 버리고 새 접근 시작 전

❌ 압축 보류 → 지금은 아니다
   - 구현 도중 (함수 절반 작성, 리팩토링 중간)
   - 테스트가 RED 상태
   - 디버깅 스택 추적 중
   - 서브에이전트 응답 대기 중
```

## 실행 프로토콜

### Phase 1: 압축 전 체크포인트 생성

```markdown
## 체크포인트 [YYYY-MM-DD HH:MM]

### 완료된 것
- [완료 항목 요약]

### 현재 상태
- 진행 중인 태스크: [태스크명]
- 현재 파일: [파일 경로]
- 마지막 결정: [중요한 결정 사항]

### 다음 할 일
- [ ] [다음 태스크 1]
- [ ] [다음 태스크 2]

### 열린 문제
- [해결되지 않은 질문/이슈]

### 핵심 컨텍스트 (압축 후 복원 필요)
- 인터페이스 정의: [핵심 인터페이스]
- 환경 변수: [필요한 설정]
- 경로: [중요한 파일 경로들]
```

### Phase 2: 보존 대상 선별

```python
# 압축 후에도 반드시 유지되어야 하는 것
MUST_PRESERVE = [
    "현재 태스크의 Spec",
    "인터페이스 계약 (함수 시그니처, 타입)",
    "실패 중인 테스트의 에러 메시지",
    "다음 단계 체크리스트",
    "중요한 아키텍처 결정",
]

# 압축 시 제거 가능한 것
CAN_REMOVE = [
    "완료된 태스크의 전체 과정",
    "이미 해결된 에러의 전체 스택",
    "탐색 중 읽었지만 사용 안 한 파일",
    "반복된 중간 결과물",
]
```

### Phase 3: 압축 실행

```bash
# Claude Code 압축 명령
/compact

# 압축 프롬프트 (자동 생성)
"다음 내용을 반드시 유지하며 압축해주세요:
[Phase 2에서 선별한 MUST_PRESERVE 항목]"
```

### Phase 4: 압축 후 검증

```
압축 완료 후 확인:
□ 체크포인트의 "다음 할 일" 목록이 보이는가?
□ 현재 태스크 Spec이 유지되는가?
□ 실패 중인 테스트 정보가 있는가?
□ 컨텍스트가 20% 이하로 감소했는가?

→ 모두 통과 시: 작업 재개
→ 실패 항목 있을 시: 체크포인트에서 수동 복원
```

## 압축 품질 지표

```
좋은 압축:
✅ 압축 후 컨텍스트 < 20%
✅ 다음 태스크 즉시 재개 가능
✅ 핵심 인터페이스 정보 유지
✅ 진행 상태 파악 가능

나쁜 압축:
❌ "방금 뭘 하고 있었더라?" 상태
❌ 파일 경로를 다시 찾아야 하는 상태
❌ 인터페이스를 다시 읽어야 하는 상태
```
